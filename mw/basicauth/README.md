# Basic Auth Middleware

## What it does

Authenticates requests using HTTP Basic authentication. It supports:

- Existing single-user setup with `basicauth.New(username, password)`.
- Multiple static users.
- Memory, CSV, JSON, or custom storage.
- Dynamic providers that return an array/slice of users from a function.
- Password hashing with PBKDF2-SHA256.
- Constant-time credential comparison.
- TLS enforcement and custom unauthorized handlers.

## How to implement

### Existing single-user usage

```go
app.Use(basicauth.New("admin", "secret"))
```

### Multiple static users from plaintext passwords

```go
mw, err := basicauth.NewFromPlainUsers(map[string]string{
    "admin": "secret",
    "ops":   "deploy-secret",
})
if err != nil {
    panic(err)
}
app.Use(mw)
```

### Multiple users with hashed passwords

```go
hash, _ := basicauth.HashPassword("secret")

app.Use(basicauth.NewFromUsers([]basicauth.User{
    {
        Username:     "admin",
        PasswordHash: hash,
        Enabled:      true,
        Roles:        []string{"admin"},
    },
}))
```

### Dynamic function returning an array of users

Use `UsersProvider` when credentials are already hashed or loaded from a production store.

```go
app.Use(basicauth.NewWithConfig(basicauth.Config{
    UsersProvider: func() ([]basicauth.User, error) {
        // Load from DB, cache, secrets manager, BCL config, file watcher, etc.
        hash, _ := basicauth.HashPassword("secret")
        return []basicauth.User{
            {Username: "admin", PasswordHash: hash, Enabled: true, Roles: []string{"admin"}},
            {Username: "ops", PasswordHash: hash, Enabled: true, Roles: []string{"ops"}},
        }, nil
    },
}))
```

### Startup-loaded plaintext users

This is convenient for tests or small deployments using mounted secrets. Load
once during startup so request handling never reads process environment. For
persistent storage, prefer hashed `UsersProvider`.

```go
adminPassword, err := config.RequireSecretString("ADMIN_PASSWORD", "ADMIN_PASSWORD_FILE")
if err != nil { log.Fatal(err) }
opsPassword, err := config.RequireSecretString("OPS_PASSWORD", "OPS_PASSWORD_FILE")
if err != nil { log.Fatal(err) }

auth, err := basicauth.NewFromPlainUsers(map[string]string{
    "admin": adminPassword,
    "ops":   opsPassword,
})
if err != nil { log.Fatal(err) }
app.Use(auth)
```

### Custom storage

```go
type DBUsers struct{}

func (DBUsers) GetUser(username string) (basicauth.User, bool, error) {
    // Query user by username and return PasswordHash generated by basicauth.HashPassword.
    return basicauth.User{}, false, nil
}

app.Use(basicauth.NewWithConfig(basicauth.Config{
    Storage: DBUsers{},
}))
```

### Access authenticated principal

```go
app.Use(basicauth.NewWithConfig(basicauth.Config{
    UsersProvider: loadUsers,
    OnAuthenticated: func(c fh.Ctx, p basicauth.Principal) error {
        c.Locals("basic_auth_user", p.Username)
        c.Locals("basic_auth_roles", p.Roles)
        return nil
    },
}))
```

## Impact

Basic Auth has low overhead with static memory users. Dynamic providers run on each request, so use caching or efficient lookups when the provider reads from external systems.

## Ordering guidance

Run early before protected handlers. Run after trusted proxy/TLS detection middleware if `RequireTLS` uses `X-Forwarded-Proto` or similar headers.

## Production considerations

- Use only over HTTPS.
- Prefer `UsersProvider` with hashed passwords for persistent stores.
- Use `PlainUsersProvider` only when secrets are already protected externally or in tests.
- Enable `RequireTLS` in production when the app can reliably detect HTTPS.
- Pair with rate limiting or abuse protection to reduce brute-force risk.
- Keep provider functions fast; cache database/secrets-manager lookups where appropriate.
